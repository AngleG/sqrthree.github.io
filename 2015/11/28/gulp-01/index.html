前端开发的工业时代

  • gulp
技术

随着 Web 技术的发展，我们前端工程师需要维护的代码也变得越来越为庞大和复杂，代码维护、压缩、打包、发布等流程也变得极为繁琐，以至于在重复劳动的过程中浪费了大量时间和精力。本着 任何花费超过90s的事情都要自动化的原则，我们要学着偷懒，啊不，学着拥抱变化，来提高效率，节约成本。我们期待有一种工具可以取代这种重复劳动。

0、为什么要用构建工具？

在日常的开发过程中，实际上我们是做了很多”重复工作”的，比如压缩、合并 CSS/JS文件，又或者你使用了诸如 less sass 这种高级工具，我们最终需要将他们编译为 CSS 文件，再比如为了解决缓存问题给文件添加 MD5 戳之类的。这些工作都是有规律但是枯燥的，在之前，我们不得不人工的去进行这些操作，耗时耗力，但是有了自动化构建工具，我们只需要找到其中的规律，然后编写一个任务，剩下的，就让工具去做吧。你所需要做的，就是喝杯咖啡，静静地等待它完成就好了。

可以说，构建工具的出现，让前端立马告别刀耕火种的苦逼时代，迈向工业化。从此，你的前端团队可以专心于业务研发，不再过多操心系统运维、代码耦合、性能优化等等，真正提高生产力。

一名合格的前端开发工程师应该至少会使用一种构建工具。

1、构建工具有哪些？

Node 发布以来，各种工具如雨后春笋般涌现而来，这其中不乏专门用来提高前端生产力的工具。本系列所要说的 Gulp 就是基于 Node 开发而来的。实际上在 Gulp 出现之前，还出现过一个叫做 Grunt 的工具，它的目的和作用和 Gulp 是一样的，但是由于基于任务的设计模式导致任务配置太过于复杂，学习曲线略微陡峭，而 Gulp 是基于流 的理念，简单直接，学习成本很低，所以一出生，就直接盖过了 Grunt 的风头。

目前业界也出现了其他的一些类似的工具，如: browserify、webpack、fis 等。这些都各有各的特点，感兴趣的可自行了解。

我们团队之前一直在使用 fis，项目一开始的时候，fis 稍微配置下就可以很方便的使用。但是随着项目的进展，渐渐的 fis 越来越不能满足项目的需求，而且 fis 的各种插件开发非常滞后。然后我们在前不久决定转换到 Gulp 了。

在学习使用 Gulp 的过程中也遇到了一些问题，因此决定写一个系列来分享一下我对 Gulp 的学习和理解。

2、如何安装 Gulp?

2.1、安装 Node 环境

因为 Gulp 是基于 Node 进行开发的，因此如果你要使用 Gulp，就必须先要确保有 Node 环境。安装 Node 的方式为:

  • windows 用户: 到 Node 官网下载对应版本的 msi 安装包，然后一键安装。
  • mac 用户: 可以去官网使用 msi 安装包，或者使用 brew install node --without-npm 命令安装 Node，然后使用 curl -L https://npmjs.org/install.sh | sh 命令安装 Npm

3、使用 Gulp

3.1、在项目中安装 Gulp

可能你已经注意到上面好像已经安装过 Gulp 了，为什么这里还要再装一次？因为上面执行的全局安装，就是为了你能够在命令行中使用 Gulp，这一次是在你的项目目录中(本地)安装，为的是在配置任务时能够调用 Gulp 来做一些操作。在项目目录执行 npm install gulp 来进行本地安装。

如果你之前没接触过 Node，则建议你先(在项目目录)进行以下操作:

  • npm init : 初始化一个 package.json 文件，按照命令的提示依次输入信息即可，该文件用来标记项目的信息以及包依赖项，便于在分享的时候只需要发送 package.json 文件，而不是所有的依赖包。
  • npm install gulp --save: 本地安装 Gulp 依赖，--save 参数能让程序自动把 Gulp依赖添加到上一步所生成的 package.json 文件中。

3.2、创建配置文件

Gulp 命令在执行任务时，会先在你的项目根目录寻找一个 gulpfile.js 的文件，并以其中的内容作为任务来进行执行操作。因此，如果你需要进行构建任务，需要在根目录创建一个 gulpfile.js 文件，并在里面编写你所需要的任务。
